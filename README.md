# pharos-nft-skill

Production-grade NFT toolkit for the Pharos blockchain, packaged as a Claude Code / OpenClaw / Codex skill. Extends the official [`pharos-skill-engine`](https://github.com/PharosNetwork/pharos-skill-engine) with the one major capability it does not cover: NFTs (ERC-721 and ERC-1155).

## What This Skill Does

When installed, the host Agent gains the ability to handle natural-language NFT questions on the Pharos chain without writing any code. The skill itself is pure Markdown plus JSON config; all on-chain work happens through Foundry (`cast`, `forge`) commands the Agent generates from these templates, batched through Multicall3.

| ERC-1155 balance and `balanceOfBatch` | `cast call balanceOf(address,uint256)` | [ownership.md](references/ownership.md) |

| Wallet holdings enumeration (log-scan fallback) | `eth_getLogs` chunked Transfer scan | [snapshot.md](references/snapshot.md) |
| Full collection holder snapshot at block N | Chunked Transfer-event replay | [snapshot.md](references/snapshot.md) |
| Single-tokenId transfer history | `eth_getLogs` filtered on topic3 | [snapshot.md](references/snapshot.md) |
| Wallet-to-wallet holdings diff | jq set ops over two enumerations | [snapshot.md](references/snapshot.md) |
| Multicall3 batch primitive | `aggregate3` template + chunking strategy | [batch.md](references/batch.md) |
| Declarative eligibility rule DSL (AND / OR / NOT, count thresholds, trait filters) | jq-evaluated rule tree | [eligibility.md](references/eligibility.md) |

### Why This Skill Exists

The official `pharos-skill-engine` covers ERC-20 and generic contract interactions but ships zero NFT capabilities. Naive NFT workflows generated by an Agent without this skill suffer from:

- N independent RPC calls instead of one Multicall3 batch.
- Missed ERC-165 detection, leading to wrong code paths on non-Enumerable collections.
- Silent failures on non-standard contracts (CryptoPunks-style).
- Wrong ownership results when NFTs are staked or held in escrow proxies.
- No primitive for "is wallet X eligible for this airdrop?": the most common multi-wallet NFT question.

This skill closes all of those gaps with the same style and structure as `pharos-skill-engine`, so Agents that already know how to use that skill can pick this one up with no friction.

## Installation

Pick the entry that matches your Agent:

| Agent | Install Command |
|-------|-----------------|
| Claude Code | `npx skills add https://github.com/iwbinb/pharos-nft-skill -g --yes` |
| OpenClaw | `npx skills add https://github.com/iwbinb/pharos-nft-skill -g --yes` |
| Codex | `npx skills add https://github.com/iwbinb/pharos-nft-skill -g --yes` |

The `-g` flag forces a global (user-level) install at `~/.agents/skills/` with symlinks into each agent's skill directory (e.g. `~/.claude/skills/pharos-nft-skill` for Claude Code). Omit `-g` if you want a project-local install at `./.agents/skills/` instead, which keeps the skill scoped to the current project.

After installation, verify it loaded:

| Agent | Verify |
|-------|--------|
| Claude Code | type `/skills` in any new conversation |
| OpenClaw | `openclaw skills list` |
| Codex | type `/skills` at session start |

You should see `pharos-nft-skill` in the list with a green checkmark.

## Prerequisites

- **Foundry** (`cast`, `forge`): `curl -L https://foundry.paradigm.xyz | bash && foundryup`
- **jq** and **curl**: ship by default on macOS and Linux

No private key needed for any of the core read-only workflows. Private key is only required if you choose to deploy the optional demo fixtures under [`assets/fixtures/`](assets/fixtures/).

## Usage Examples

After install, just talk to the Agent in natural language. The skill triggers automatically on NFT-related Pharos questions.

```text
> Does wallet 0xabc...123 own any NFTs from collection 0xdef...456 on Pharos testnet?
> List every tokenId in 0xdef...456 held by 0xabc...123, with metadata.
> Who currently owns tokenId 42 in collection 0xdef...456?
> Give me a holder snapshot of collection 0xdef...456 at block 22000000.
> Reconstruct the full transfer history of tokenId 42 in 0xdef...456.
> Compare the NFT holdings of 0xabc...123 and 0xfff...000. Which tokens are unique to each?
> Run eligibility check from rule.json against the wallets in candidates.txt.
```

The Agent reads [SKILL.md](SKILL.md) to identify which reference file to load, generates the appropriate `cast` / `forge` commands, and runs them against the configured Pharos RPC.

## Design Principles

This skill is opinionated about a handful of things that matter when running NFT workflows against a public RPC at scale:

1. **Read-first**. Nothing in the core capability set requires a private key. Gating and airdrop workflows should be safe to run with read-only RPC access.
2. **No explorer scraping**. Pharos's explorer has anti-bot protection. The skill relies on RPC + Multicall3 + event logs, never on HTML parsing.
3. **Multicall3 by default**. Any workflow that would otherwise issue 4+ sequential `cast call` invocations is folded into one Multicall3 batch. Saves both wall time and rate-limit pressure.
4. **Bounded event scans**. `eth_getLogs` ranges chunked to `logScanMaxBlocks` (default 10,000) to stay inside RPC limits, never assuming an unlimited range.
5. **Graceful degradation on metadata**. Eligibility checks complete even when IPFS gateways time out; metadata is treated as optional context, never a blocker.
6. **One declarative DSL for all eligibility logic**. A 4-node rule schema (`all_of` / `any_of` / `none_of` / `min_count`) covers every airdrop snapshot, whitelist, and gating use case we've seen.

## Testing

The repo ships a self-contained test suite that needs only bash, `jq`, `python3`, and `curl` (no Node, no Foundry, no compiler). See [tests/README.md](tests/README.md) for the full description.

```bash
# Lint + jq (no network)
bash tests/run.sh

# Lint + jq + live RPC against Pharos Atlantic testnet
bash tests/run.sh --with-live
```

Three suites, 16 tests:

- **lint** (6 tests): em/en-dash forbid, JSON validity, frontmatter shape, markdown link resolution, anchor resolution, and `bash -n` syntax check on every fenced code block in the docs.
- **jq** (4 tests): 15 fixture cases of the eligibility-DSL evaluator (AND / OR / NOT / nested), 8 trait-match cases, holdings-diff verification, doc-evaluator drift check.
- **live** (6 tests, requires `--with-live`): RPC reachability, Multicall3 bytecode presence, `aggregate3([])` round trip, `eth_call balanceOf` shape, `eth_getLogs` shape, and a hand-encoded `aggregate3` single-call sanity check.

A real bug surfaced in the eligibility DSL evaluator during the first run of the jq suite (filter-parameter vs value-parameter confusion in jq's `def f(r; h)` syntax, which caused recursive lookups to evaluate against the wrong `.`). The fix is documented inline in `references/eligibility.md` so future readers do not re-introduce it.

## Composing With pharos-skill-engine

Install both skills together. They share:

- the same `networks.json` shape (this skill ships its own copy so it works standalone),
- the same Foundry toolchain,
- the same Pre-check pattern for write operations,
- the same reference-section structure (Command Template → Parameters → Output Parsing → Error Handling → Agent Guidelines).

An Agent that has both installed can answer compound questions like "swap 100 USDC for the NFT at tokenId 42 and check the holder afterwards" by routing the swap to `pharos-skill-engine` and the holder check to `pharos-nft-skill` without manual hand-off.

## Verifying Multicall3 Is Live on Pharos

This skill assumes Multicall3 is deployed at the canonical address `0xcA11bde05977b3631167028862bE2a173976CA11` on Pharos. Verified live on Atlantic testnet:

```bash
curl -s https://atlantic.dplabs-internal.com \
  -H 'Content-Type: application/json' \
  --data '{"jsonrpc":"2.0","method":"eth_getCode","params":["0xcA11bde05977b3631167028862bE2a173976CA11","latest"],"id":1}'
```

A non-empty `result` confirms the contract is deployed.
